| 门禁 | 位置 | 提交物 | 审批人 | 通过标准 | 未通过回退 |

## 阶段一：需求分析

### 步骤1：需求交底

| 项目 | 内容 |
|------|------|
| **触发条件** | 产品组发起需求交底会议 |
| **RACI** | R：产品组 / A：后端研发组 / I：项目经理 / C：前端厂商 |
| **关键活动** | 1. 产品组宣讲需求背景、业务目标、用户场景<br>2. 后端研发组提问澄清需求边界与约束<br>3. 明确需求优先级与交付时间节点 |
| **交付物** | 需求交底会议纪要（含需求清单、优先级、时间节点） |
| **准入条件** | 产品组提供需求文档/PRD |
| **注意事项** | 后端研发组须在会上识别技术风险并记录，避免后期返工 |

### 步骤2：后端功能分析

| 项目 | 内容 |
|------|------|
| **RACI** | R：后端研发组 / A：技术负责人 / I：产品组 / C：项目经理 |
| **关键活动** | 1. 将需求拆解为后端功能点<br>2. 识别技术依赖（第三方服务、数据源、基础设施）<br>3. 评估技术可行性与风险点<br>4. 确定技术方案选型 |
| **交付物** | 后端功能分析文档（功能点清单、技术方案、风险清单） |
| **注意事项** | 涉及跨系统交互的功能点须在此阶段明确调用关系 |

### 步骤3：接口拆解

| 项目 | 内容 |
|------|------|
| **RACI** | R：后端研发组 / A：前端厂商 / I：产品组 |
| **关键活动** | 1. 根据功能点梳理前后端交互接口清单<br>2. 明确每个接口的请求方式、业务含义<br>3. 标注接口间的依赖关系与调用顺序<br>4. 区分新增接口与复用接口 |
| **交付物** | 接口清单（接口名称、请求方式、业务描述、依赖关系） |
| **注意事项** | 接口清单须与前端厂商对齐，确保前后端理解一致 |

### 步骤4：工作量评估

| 项目 | 内容 |
|------|------|
| **RACI** | R：后端研发组 / A：项目经理 / I：产品组 / C：前端厂商 |
| **关键活动** | 1. 按接口/功能点逐项评估开发工时<br>2. 预留联调、自测、文档编写时间<br>3. 识别关键路径与并行开发可能性<br>4. 输出排期计划 |
| **交付物** | 工作量评估表（接口级工时、总工时、排期计划） |
| **注意事项** | 工时评估须包含缓冲时间（建议10%-15%），复杂接口单独标注风险 |

## 🔶 评审门禁①：接口清单 + 工作量预估

| 项目 | 内容 |
|------|------|
| **提交物** | 接口清单 + 工作量评估表 |
| **审批人** | 产品组（确认需求覆盖度）、项目经理（确认排期合理性） |
| **通过标准** | 1. 接口清单覆盖所有需求功能点<br>2. 工作量评估无重大遗漏<br>3. 排期与项目里程碑对齐 |
| **未通过处理** | 返回步骤2（后端功能分析），重新梳理功能点与接口 |
| **强化要求** | 接口清单须前端厂商确认签收，避免后续接口理解偏差 |

## 阶段二：接口设计

### 步骤5：接口协议编写

| 项目 | 内容 |
|------|------|
| **RACI** | R：后端研发组 / A：前端厂商 / C：技术负责人 |
| **关键活动** | 1. 编写每个接口的详细协议文档<br>2. 定义请求参数（字段名、类型、必填、校验规则）<br>3. 定义响应结构（成功响应、错误码、错误信息）<br>4. 定义公共规范（分页、鉴权、版本号、时间格式） |
| **交付物** | 接口协议文档（每个接口含：URL、Method、请求参数、响应结构、错误码、示例） |
| **注意事项** | ⚠️ **痛点强化**：<br>1. 错误码须统一编码，禁止各接口自定义<br>2. 枚举值须提供完整字典<br>3. 金额、时间等字段须明确精度与格式<br>4. 接口变更须记录变更日志 |

### 步骤6：接口协议评审

| 项目 | 内容 |
|------|------|
| **RACI** | R：后端研发组 / A：前端厂商 / I：产品组、测试组 / C：技术负责人 |
| **关键活动** | 1. 后端逐接口讲解协议设计<br>2. 前端厂商逐接口确认可消费性<br>3. 测试组确认接口可测试性<br>4. 记录评审问题与修改项 |
| **交付物** | 接口协议评审记录（含问题清单、修改项、确认结果） |
| **注意事项** | ⚠️ **痛点强化**：<br>1. 评审须前后端双方**现场/线上会议**进行，禁止仅文档异步评审<br>2. 评审后接口协议冻结，变更须走变更流程<br>3. 前端厂商须在评审记录上签字/确认 |

## 🔶 评审门禁②：接口协议 + 前后端评审

| 项目 | 内容 |
|------|------|
| **提交物** | 接口协议文档 + 评审记录 |
| **审批人** | 前端厂商（确认接口可消费）、后端技术负责人（确认技术方案）、项目经理（确认无阻塞） |
| **通过标准** | 1. 前后端双方对全部接口达成一致<br>2. 评审问题全部闭环<br>3. 接口协议版本冻结 |
| **未通过处理** | 返回步骤5（接口协议编写），修改后重新评审 |
| **强化要求** | 1. 评审记录须有双方签字/确认截图<br>2. 冻结后的接口变更须提交变更申请并重新评审<br>3. Mock数据须在评审通过后3个工作日内提供 |

## 阶段三：开发与联调

### 步骤7：开发

| 项目 | 内容 |
|------|------|
| **RACI** | R：后端研发组 / A：技术负责人 / C：前端厂商 |
| **关键活动** | 1. 按接口协议编码实现<br>2. 编写单元测试（核心逻辑覆盖率≥80%）<br>3. 提供Mock数据供前端并行开发<br>4. 代码提交前完成自测 |
| **交付物** | 后端代码（含单元测试）、Mock数据 |
| **注意事项** | 1. 开发过程中发现接口协议需变更，须走变更流程，禁止私自修改<br>2. Mock数据须与接口协议一致 |

### 步骤8a：研发负责人接口测试

| 项目 | 内容 |
|------|------|
| **RACI** | R：研发负责人 / A：后端研发组 / I：项目经理 |
| **关键活动** | 1. 按接口协议逐接口验证实现完整性<br>2. 覆盖不同业务场景（正常流、异常流、边界值）<br>3. 校验响应字段完整性（上下文字段、关联字段无缺失）<br>4. 校验错误码与错误信息与协议一致<br>5. 发现问题打回开发修复，修复后重新验证<br>6. **通过测试的接口增量交付给8b联调** |
| **交付物** | 接口测试报告（接口级测试结果、问题清单、修复确认） |
| **注意事项** | 🚨 **核心守门环节**：<br>1. 研发负责人是接口提交质量的唯一守门人，未通过接口测试的接口不得进入前后端联调<br>2. 重点检查：字段缺失、类型不符、枚举值遗漏、空值处理、分页参数<br>3. 接口测试不通过，开发人员须修复后重新提交，研发负责人重新验证<br>4. **先行启动，开发数日后增量交付已通过接口给8b联调** |

### 步骤8b：前后端联调

| 项目 | 内容 |
|------|------|
| **RACI** | R：后端研发组 / A：前端厂商 / I：测试组 |
| **关键活动** | 1. 接收8a增量交付的已通过接口，按接口协议逐接口对接<br>2. 记录联调问题（字段不符、数据格式、边界情况）<br>3. 修复联调缺陷<br>4. 全部接口联调通过 |
| **交付物** | 联调问题清单及修复记录、联调通过确认 |
| **注意事项** | 1. 联调环境须与接口协议版本一致<br>2. 联调问题须当日记录、当日修复或给出修复计划<br>3. **仅接收8a增量交付的已通过接口测试的接口**<br>4. **联调发现的阻断性问题反馈给后端研发持续修复支持** |

> **步骤8协作说明**：8a（研发负责人接口测试）先行启动，开发数日后将已通过测试的接口增量交付给8b（前后端联调）。8b联调过程中发现的阻断性问题反馈给后端研发组持续修复支持，形成 **8a增量交付 → 8b联调 → 反馈修复 → 增量交付** 的迭代循环。两个环节均完成后，方可进入评审门禁③。

## 🔶 评审门禁③：后端测试报告

| 项目 | 内容 |
|------|------|
| **位置** | 步骤8（开发联调）完成后 → 步骤9（测试组测试）之前 |
| **提交物** | 后端测试报告（含接口测试结果、联调结果、缺陷统计、遗留缺陷清单） |
| **提交人** | 研发负责人 |
| **审批人** | 项目经理（确认可提交测试组） |
| **通过标准** | 1. 研发负责人接口测试全部通过<br>2. 前后端联调全部通过<br>3. 无P0/P1遗留缺陷<br>4. P2及以下缺陷有明确修复计划 |
| **未通过处理** | 返回步骤7（开发），修复缺陷后重新走8a接口测试 + 8b联调 |
| **核心意义** | 后端质量守门——未通过此门禁的代码不得进入测试组，减少测试组返工 |

## RACI 矩阵总览

> **R** = Responsible（执行） / **A** = Accountable（审批/负责） / **I** = Informed（知会） / **C** = Consulted（咨询）

| 步骤 | 后端研发组 | 研发负责人 | 前端厂商 | 产品组 | 测试组 | 项目经理 | 运维/DevOps |
|------|-----------|-----------|---------|--------|--------|---------|------------|
| 1.需求交底 | A | - | C | **R** | I | I | - |
| 2.后端功能分析 | **R** | A | I | C | - | I | - |
| 3.接口拆解 | **R** | A | C | I | - | I | - |
| 4.工作量评估 | **R** | A | C | I | - | I | - |
| 5.接口协议编写 | **R** | C | A | - | C | I | - |
| 6.接口协议评审 | **R** | C | A | I | I | C | - |
| 7.开发 | **R** | A | C | - | - | I | - |
| 8a.研发负责人接口测试 | C | **R** | - | - | - | I | - |
| 8b.前后端联调 | **R** | - | A | - | I | C | - |
| ③后端测试报告 | C | **R（提交）** | - | - | - | **A（审批）** | - |

### OPA-2026-001：核心字段上下文贯通

| 项目 | 内容 |
|------|------|
| **问题分类** | 接口设计 |
| **关联步骤** | 步骤5（接口协议编写）、步骤6（接口协议评审）、步骤8a（研发负责人接口测试） |
| **问题描述** | 不同接口之间上下文未贯通，导致前端无法串联业务流程。典型场景：详情接口未返回修改提交接口所需的核心字段（如数据主键、唯一索引），前端需额外发起查询或硬编码拼接 |
| **影响** | 1. 前端需额外请求补全数据，增加交互轮次<br>2. 前端自行拼接关键标识，易出错且难维护<br>3. 接口联调时反复返工补充字段 |
| **规范要求** | 1. **查询类接口必须返回数据主键或唯一索引**，确保前端可基于返回结果直接发起后续操作<br>2. 接口协议编写时须标注"上下游接口关联"，明确当前接口的返回字段被哪些下游接口消费<br>3. 接口协议评审时须检查上下游接口字段衔接完整性<br>4. 研发负责人接口测试时须验证核心字段贯通性 |
| **检查清单** | ☐ 详情/列表接口是否返回主键或唯一索引<br>☐ 下游接口所需字段是否在上游接口返回中覆盖<br>☐ 关联接口的字段命名是否一致 |

### OPA-2026-002：相同业务场景数据结构一致性

| 项目 | 内容 |
|------|------|
| **问题分类** | 接口设计 |
| **关联步骤** | 步骤5（接口协议编写）、步骤6（接口协议评审） |
| **问题描述** | 相同业务含义的详情/列表接口，返回的数据结构不一致（字段命名不同、嵌套层级不同、数据类型不同），导致前端需要针对同一业务对象编写多套适配逻辑 |
| **影响** | 1. 前端代码冗余，同一业务对象需多套数据映射<br>2. 联调阶段频繁因格式不一致返工<br>3. 后续维护成本高，修改一处需同步多处 |
| **规范要求** | 1. **同一业务对象在不同接口中的数据结构必须一致**，包括字段命名、类型、嵌套结构<br>2. 接口协议编写时须定义公共数据模型（DTO），各接口引用公共模型而非各自定义<br>3. 接口协议评审时须比对相同业务对象的数据结构一致性 |
| **检查清单** | ☐ 同一业务对象在不同接口中字段命名是否一致<br>☐ 同一业务对象在不同接口中数据类型是否一致<br>☐ 是否已定义公共数据模型并引用 |

### OPA-2026-003：接口合并策略

| 项目 | 内容 |
|------|------|
| **问题分类** | 接口设计 |
| **关联步骤** | 步骤3（接口拆解）、步骤5（接口协议编写） |
| **问题描述** | App客户端场景下，接口拆分过细导致请求次数过多，页面加载慢、用户体验差。合理场景下应将关联查询合并为单一接口，减少网络往返 |
| **影响** | 1. 页面加载耗时增加，多个串行请求导致白屏时间长<br>2. 移动端网络不稳定时，请求失败概率增大<br>3. 前端需自行管理多个请求的并发与依赖 |
| **规范要求** | 1. **App客户端场景优先考虑接口合并**，将页面所需的关联子查询合并到主查询接口中<br>2. 合并原则：同一页面、同一时序、无独立复用价值的子查询应合并<br>3. 不合并原则：有独立复用价值的查询、不同页面使用的查询、数据量差异过大的查询不宜合并<br>4. 接口拆解阶段须明确标注"App端需合并"的接口，并在接口协议中体现合并后的结构 |
| **检查清单** | ☐ App端页面是否需要多次请求才能加载完成<br>☐ 子查询是否有独立复用价值<br>☐ 合并后接口响应体是否过大（建议单接口响应<100KB） |

### OPA-2026-004：码表一致性

| 项目 | 内容 |
|------|------|
| **问题分类** | 接口设计 |
| **关联步骤** | 步骤5（接口协议编写）、步骤6（接口协议评审）、步骤8a（研发负责人接口测试） |
| **问题描述** | 多个接口返回的相同码表（如状态码、类型码、字典项）值不一致，同一业务含义在不同接口中返回不同的编码或标签，导致前端展示混乱 |
| **影响** | 1. 前端需维护多套码表映射，增加复杂度<br>2. 同一状态在不同页面展示不一致，用户困惑<br>3. 码表变更时需同步修改多处，易遗漏 |
| **规范要求** | 1. **全局码表统一定义、统一维护**，禁止各接口自行定义码表值<br>2. 接口协议中引用码表须标注码表来源（全局码表ID）<br>3. 码表变更须走变更流程，评估影响范围<br>4. 研发负责人接口测试时须校验码表一致性 |
| **检查清单** | ☐ 是否存在同一码表在不同接口中值不同的情况<br>☐ 码表是否有统一定义和维护入口<br>☐ 接口协议中是否标注了码表来源 |

### OPA-2026-005：接口拆分粒度控制

| 项目 | 内容 |
|------|------|
| **问题分类** | 接口设计 |
| **关联步骤** | 步骤3（接口拆解）、步骤5（接口协议编写） |
| **问题描述** | 后端将不同状态的查询拆解为多个独立接口（如：待审核列表、已通过列表、已驳回列表），每个接口返回结构部分不同。这种拆分方式导致前端需调用不同接口、维护多套逻辑，后端也存在大量重复代码 |
| **影响** | 1. 前端需根据状态判断调用不同接口，逻辑分散<br>2. 后端多个接口存在重复查询逻辑，维护成本高<br>3. 新增状态时需新增接口，扩展性差 |
| **规范要求** | 1. **同一业务对象的不同状态查询应合并为单一接口**，通过请求参数（如status字段）区分状态<br>2. 返回结构中不同状态特有的字段可置空或省略，但公共结构必须一致<br>3. 仅当不同状态的查询逻辑差异极大（如不同数据源、不同权限模型）时才允许拆分，且须在接口协议中说明拆分理由<br>4. 接口拆解阶段须评估拆分合理性，避免过度拆分 |
| **检查清单** | ☐ 是否存在仅状态不同就拆分为多个接口的情况<br>☐ 合并后是否可通过参数区分状态<br>☐ 不同状态的公共字段结构是否一致 |

### OPA-2026-006：异常编码统一规范

| 项目 | 内容 |
|------|------|
| **问题分类** | 接口设计 |
| **关联步骤** | 步骤5（接口协议编写）、步骤6（接口协议评审）、步骤8a（研发负责人接口测试） |
| **问题描述** | 接口异常值编码不统一，系统异常与业务异常未区分，前端无法根据错误码判断异常类型并采取不同处理策略（如业务异常提示用户、系统异常引导重试或联系运维） |
| **影响** | 1. 前端无法区分异常类型，统一弹"系统异常"提示，用户体验差<br>2. 业务异常（如余额不足）与系统异常（如数据库超时）混在一起，排查困难<br>3. 异常处理逻辑散落各处，无法统一拦截 |
| **规范要求** | 1. **必须区分系统异常与业务异常**，编码规则须在接口协议公共规范中定义<br>2. 推荐编码规则：<br>　　- 系统异常：`SYS-XXXX`（如 SYS-5001 数据库超时、SYS-5002 服务不可用）<br>　　- 业务异常：`BIZ-XXXX`（如 BIZ-1001 余额不足、BIZ-1002 订单不存在）<br>3. 异常响应结构统一：<br>　　```json<br>　　{ "code": "BIZ-1001", "message": "余额不足", "type": "BIZ", "traceId": "xxx" }<br>　　```<br>4. 系统异常须返回traceId便于日志追踪，禁止向前端暴露堆栈信息<br>5. 研发负责人接口测试时须覆盖异常场景并校验编码规范 |
| **检查清单** | ☐ 异常编码是否区分系统异常与业务异常<br>☐ 异常响应结构是否统一（code/message/type/traceId）<br>☐ 系统异常是否返回traceId且不暴露堆栈<br>☐ 异常编码是否有全局注册表，避免重复 |
